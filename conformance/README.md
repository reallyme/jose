# JOSE Conformance

This directory records the conformance surface for `reallyme-jose`.

Requirement records name the normative source section, the implementation files
that enforce it, and the positive and negative tests that cover it. Vector
files are portable fixtures used by local Rust tests and by SDKs that need
byte-for-byte JOSE parity.

ES256 verification rejects malformed signatures, wrong-length signatures, and
correct-length signatures whose `r` or `s` scalar is outside the P-256 scalar
field. It does not add a JOSE-specific low-S normalization rule on top of
backend ECDSA verification; high-S signatures that are otherwise valid remain
valid ECDSA signatures. This is intentional even when a platform signer, such
as a Face ID or Secure Enclave protected P-256 key, emits a canonical low-S form:
the conformance contract is verifier acceptance of valid JOSE/P-256 ECDSA, not
platform-specific signer byte uniqueness.

The vector manifest in [vectors/manifest.json](vectors/manifest.json) is audited
by the standalone `tools/vector-audit` crate. The audit tool avoids linking the
production JOSE, codec, and crypto crates so committed fixtures are checked by
an independent implementation path.

The `panva-jose` vector suite is a small positive interop anchor generated by
`tools/panva-goldens` with `panva/jose@6.2.3`. It is intentionally limited to
JWS/JWT verification and ECDH-ES JWE decryption cases where an independent
implementation catches bugs that local round-trips can miss. The Node tool is a
maintenance generator only; normal Rust tests consume the checked-in JSON.

The checked-in panva vectors currently execute through the native Rust test
lane. CI verifies that the wasm lane compiles, but it does not yet execute these
fixtures through the wasm crypto provider. Adding a wasm fixture runner is an
additive hardening task before claiming wasm interop equivalence.

The panva suite is deliberately thin: ES256 compact JWS verification, EdDSA
compact JWS verification, ES256 compact JWT verification, and one ECDH-ES
P-256/A128GCM compact JWE decryption case. The JWE case covers the highest-risk
Concat-KDF interop path, but it is not a curve-by-content-encryption matrix.
ES256K is also outside this suite because panva/WebCrypto does not provide
secp256k1 JOSE signing support.

The panva fixture keys are deterministic low-scalar test keys so the generated
tokens remain reproducible. They are conformance fixtures only and are not
examples of production key generation.
